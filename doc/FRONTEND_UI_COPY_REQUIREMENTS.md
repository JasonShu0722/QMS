# 前端界面文案要求

## 1. 总原则

- 界面优先依靠结构、层级、标签和控件本身表达含义，不靠大段解释性文字补充理解。
- 如果一个词语本身已经足够清晰，就不要再写一行重复解释它“是做什么的”。
- 默认采用“短标题 + 直接操作”的方式组织界面，不把按钮、卡片、页签做成说明书。

## 2. 默认做法

- 按钮默认只保留动作名称，例如：`系统设置`、`保存`、`上传签名`。
- 卡片默认只保留标题，不额外补“查看 / 进入 / 管理 / 配置”这类重复描述。
- 页签默认只保留页签名，不在页签内容顶部重复写一段同义说明。
- 弹窗默认直接进入表单或内容区，除非存在高风险或高门槛，否则不增加引导性段落。
- 空态默认一句话解决，例如：`暂无公告`、`暂无快捷入口`，不追加背景说明。

## 3. 允许保留说明文案的场景

- 存在不可逆操作、数据删除、发布影响等风险提醒。
- 存在明确规则约束，例如密码复杂度、上传格式限制、权限影响范围。
- 用户如果看不到这句提示，就很可能做错操作。
- 法务、审计、合规要求必须展示的内容。

## 4. 明确禁止的写法

- 不要在卡片标题下重复写“进入某某页面”“管理某某功能”。
- 不要在设置弹窗顶部写大段“这里可以做什么”的介绍。
- 不要为了“看起来更丰富”而填充解释性副标题。
- 不要把用户已经能从上下文理解的信息，再换一句话解释一遍。

## 5. 交付检查项

- 新页面提交前，先检查是否存在可以删除而不影响理解的副标题、提示语、说明段落。
- 如果删除后界面仍然清楚，应直接删除。
- 如果必须保留说明，文案应缩到最短，只保留决策必需信息。

## 6. 当前项目执行要求

- 工作台、系统设置、快捷入口、后台管理页默认遵守本要求。
- 后续新增页面或重构页面时，把“减少冗余解释文案”作为固定验收项。
